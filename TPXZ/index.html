整体方案

会诊平台查看模式图片旋转优化

本页面用于向产品、研发、测试同学说明病历审核场景下图片方向调整的产品思路、交互原型和 PRD。 默认展示整体方案，右上角可切换查看本次需求涉及的原型与 PRD。

核心结论：查看模式下旋转图片默认视为修正资料方向，系统自动保存；但保存机制需从“关闭预览时保存”优化为“旋转后防抖保存最终角度”。

1. 需求背景

会诊平台病历审核老师在查看检查、检验、影像资料时，经常遇到上传图片方向不正的问题。当前系统在小图模式支持预览、编辑、下载、旋转； 大图预览使用预览组件，并在点击右上角关闭按钮时保存旋转角度。

线上问题并不在于“默认保存”这个产品逻辑，而在于保存触发时机和保存过程过重：旋转高频操作容易带来重复请求、页面刷新、状态不同步和多人覆盖。

2. 产品方案思路

默认保存，不增加操作成本

审核老师把图片转正，通常是在修正资料展示方向，而不是做个人临时查看偏好。因此不要求老师额外点击“保存”。

关闭按钮只关闭预览

关闭按钮的语义是退出查看，不应承载保存业务。保存动作改由旋转后的防抖任务自动完成。

前端即时响应

点击旋转后小图和大图立即改变角度，接口保存异步进行，避免审核阅读被打断。

版本校验防覆盖

同一图片被多人同时修改时，通过图片级 rotationVersion 判断冲突，不允许静默覆盖他人的调整。

点击旋转→ 即时展示→ 防抖 1 秒→ 保存最终角度→ 同步状态

3. 方案概览

模块 改造点 原则
小图模式 点击旋转后缩略图立即转动，显示保存状态，停止操作后自动保存。 轻操作、即时反馈。
大图预览 工具栏旋转后即时展示，关闭按钮不再保存，仅关闭预览。 组件负责查看，业务层负责保存。
保存机制 同一图片连续旋转合并为一次请求，仅提交最终角度；成功只在图片卡片内弱提示。 减少请求，无刷新，不用双重提示打扰审核。
多人冲突 保存时携带图片方向版本号，版本不一致则提示用户先查看最新图片。 不静默覆盖，先看最新再调整。

4. 多人同时打开的处理

图片旋转采用轻量乐观锁。用户打开病历时获取图片当前角度和版本号；保存时提交 baseRotationVersion。 若服务端版本仍一致则保存成功；若版本已变化，说明其他用户已经修改过，当前用户收到冲突提示。

冲突文案建议：该图片方向已被其他老师先一步保存。请先查看最新图片方向，再判断是否还需要继续调整。

5. 版本规则与产品边界

规则 产品定义 技术实现建议
版本对象 版本控制到单张图片，不锁整份病历。 每张图片维护 rotationVersion。
保存校验 用户保存时必须基于自己看到的图片方向版本。 保存请求携带 baseRotationVersion。
冲突处理 若别人已先保存，不允许当前用户静默覆盖；先引导查看最新图片。 后端返回 ROTATION_VERSION_CONFLICT 和最新图片方向数据。
再次调整 用户查看最新图片后，如果仍认为方向不对，可以再次旋转保存。 前端用最新 rotationVersion 发起新的保存请求。
产品经理需要定义冲突体验、用户提示、是否允许再次覆盖、日志要求和状态边界；技术同学基于这些产品规则设计字段、接口、并发校验和异常码。

PRD

查看模式图片旋转自动保存优化 PRD

本 PRD 用于明确图片旋转优化的业务目标、功能范围、交互规则、接口数据、异常处理和验收标准。

1. 背景与目标

背景

病历审核老师查看图片资料时，部分上传图片方向不正。当前旋转保存依赖关闭预览或频繁请求，影响审核效率和系统稳定性。

目标

  • 旋转后图片即时展示。
  • 系统自动保存，无需用户手动提交。
  • 连续旋转只保存最终角度。
  • 保存过程不刷新页面。
  • 多人同时操作时不静默覆盖。

2. 功能需求

编号 需求 优先级 验收说明
FR-1 小图模式点击旋转后即时更新缩略图角度。 P0 点击后 100ms 内完成前端展示，不等待接口返回。
FR-2 大图预览点击旋转后即时更新大图角度。 P0 大图与顶部缩略图角度同步。
FR-3 旋转后自动防抖保存最终角度。 P0 连续旋转多次，后端只收到最后一次角度。
FR-4 关闭预览不触发保存。 P0 关闭按钮仅关闭预览，保存由旋转后的防抖任务处理。
FR-5 展示保存状态。 P0 包含保存中、已保存、保存失败、冲突提示；成功不再额外弹全局 toast。
FR-6 支持图片级版本冲突校验。 P0 版本不一致时提示用户先查看最新图片，不静默覆盖。

3. 交互规则

  • 小图 hover 后展示预览、编辑、下载、旋转；点击旋转后图片立即转动。
  • 大图预览工具栏保留旋转、缩放、下载；点击旋转后大图立即转动。
  • 旋转后显示“保存中”，保存成功后显示“已保存”并短暂消失。
  • 关闭预览时不保存；若后台仍在保存，继续完成保存任务。
  • 保存失败时保留当前展示角度，在图片卡片内显示“保存失败”和“重试”入口。

4. 数据与接口

图片资料需返回当前角度与版本号。角度只支持 0、90、180、270。

fileId rotation rotationVersion updatedBy updatedAt 

PATCH /api/cases/{caseId}/files/{fileId}/rotation

{
  "rotation": 180,
  "baseRotationVersion": 3
}
{
  "code": "ROTATION_VERSION_CONFLICT",
  "message": "图片方向已被其他用户更新",
  "latest": {
    "rotation": 90,
    "rotationVersion": 4,
    "updatedBy": "赵老师",
    "updatedAt": "2026-07-09 15:23:18"
  }
}

5. 版本与冲突规则

规则项 规则说明 开发约束
版本粒度 仅图片方向调整参与版本校验，不影响病历主流程。 以 fileId 维度维护 rotationVersion。
正常保存 用户保存时，若服务端版本与 baseRotationVersion 一致，则保存成功。 保存成功后返回新的 rotation、rotationVersion、updatedBy、updatedAt。
冲突保存 若服务端版本已变化，说明该图片方向已被他人先保存。 接口返回 ROTATION_VERSION_CONFLICT，不更新当前用户提交的角度。
冲突后的用户路径 前端提示用户先查看最新图片方向，用户查看后如仍需调整，可再次旋转保存。 查看最新后，前端必须用最新 rotationVersion 作为下一次保存基准。
日志要求 成功保存需记录操作人、原方向、新方向、版本号、操作来源。 冲突请求可记录失败日志；再次保存成功需记录新的成功日志。

6. 异常与验收

场景 处理规则 验收标准
连续旋转 防抖合并保存。 只提交最终角度。
保存失败 保留展示角度，在图片卡片内提供重试。 页面不刷新，状态不丢失，用户可对失败图片单独重试。
多人冲突 提示用户先查看最新图片方向。 不静默覆盖；查看最新后可再次旋转保存。
无权限或病历不可修改 不允许保存方向。 前端禁用或保存失败后提示原因。
非目标：本期不改造病历编辑模式文件增删改，不做实时协同，不新增裁剪、标注、图片增强等高级编辑能力。
